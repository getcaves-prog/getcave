01 Resumen Ejecutivo

Caves es una plataforma de descubrimiento de eventos basada en geolocalización, orientada al mercado mexicano (demográfico 18-35 años). Los usuarios navegan flyers de eventos a pantalla completa, ordenados por proximidad — como un mapa social visual donde los eventos flotan alrededor de tu ubicación. El MVP se construye como una PWA mobile-first con Next.js 15, Supabase y un lenguaje de diseño dark-first.

Principio clave: Cuando abrís la carpeta features/, inmediatamente entendés lo que la aplicación HACE (feed, auth, eventos, perfil) — no cómo está construida (componentes, hooks, servicios). La estructura de carpetas grita el dominio de negocio.

04 Supabase vs Firebase: Decisión Técnica

La elección de base de datos fue la decisión arquitectónica más crítica para Caves. El valor core de la app es el descubrimiento de eventos basado en geolocalización — lo que significa que las consultas geoespaciales no son un nice-to-have, SON el producto.

Supabase Elegido

  • PostGIS nativo: ST_DWithin, ST_Distance, índices GIST — consultas de proximidad sub-10ms en millones de filas
  • Potencia SQL: JOINs complejos (eventos + categorías + perfiles), agregaciones, window functions
  • Row Level Security: Políticas auth-aware a nivel de BD — sin necesidad de middleware backend
  • Open source: Sin vendor lock-in, auto-hosteable, ecosistema Postgres
  • Generación de tipos: Tipos TypeScript auto-generados desde el schema
  • Migraciones SQL reales: Versionadas, revisables, con rollback seguro
  • Costo: Free tier generoso, pricing predecible por proyecto

Firebase Rechazado

  • Sin geo nativo: Firestore requiere workaround con GeoHash — impreciso, complejo, no puede ordenar por distancia
  • Limitaciones NoSQL: Sin JOINs, sin agregaciones, denormalización requerida en todos lados
  • Reglas de seguridad: Basadas en JSON, difíciles de testear, sin enforcement a nivel SQL
  • Vendor lock: Propietario de Google, sin self-hosting, exportación de datos dolorosa
  • Sin generación de tipos: Mantenimiento manual de tipos, riesgo de drift del schema
  • Costo impredecible: Facturación por lectura/escritura que puede dispararse con uso intensivo de feed
  • Servicios innecesarios: ML, analytics, messaging que no necesitamos

El Argumento Geoespacial (Factor Decisivo)

ConsultaSupabase (PostGIS)Firebase (GeoHash)
Encontrar eventos dentro de 25kmST_DWithin(location, point, 25000) — círculo exacto, indexadoQuery por rango GeoHash — aproximación rectangular, falsos positivos, post-filtrado requerido
Ordenar por distanciaORDER BY ST_Distance(...) — nativo, indexadoNo es posible server-side — hay que traer todo, calcular en el cliente y ordenar
Filtro + geo + ordenUna sola consulta SQL con JOIN, WHERE, ORDER BYMúltiples queries + merge en cliente + orden manual — problema N+1
Performance a escalaÍndice GIST: O(log n) sin importar el tamaño del datasetGeoHash: se degrada con densidad, requiere índices compuestos por filtro

Veredicto: Para un producto location-first, PostGIS no es solo mejor — es otra categoría. Firebase requeriría construir un motor de consultas geográficas sobre una base de datos que no fue diseñada para eso. Supabase nos da los 30+ años de madurez geoespacial de PostgreSQL out of the box.

05 Arquitectura por Features — Detalle

Reglas de Aislamiento de Features

  1. Sin imports cruzados: Los features nunca importan entre sí. El código compartido vive en shared/
  2. Server Components por defecto: Solo usar "use client" cuando la interactividad lo requiere
  3. Colocación: Tests, tipos y servicios viven junto al código que sirven
  4. Sin barrel exports: Importar directamente del archivo, no desde index.ts

06 Modelo de Datos

Relación de Entidades

profiles (1) ──────< events (N) "Un usuario crea muchos eventos" categories (1) ────< events (N) "Un evento pertenece a una categoría" ┌─────────────┐ ┌──────────────────────────────────────────┐ │ profiles │ │ events │ ├─────────────┤ ├──────────────────────────────────────────┤ │ id (PK/FK) │────>│ user_id (FK) │ │ username │ │ category_id (FK) │ │ avatar_url │ │ title, description │ │ bio │ │ flyer_url (Supabase Storage) │ │ role │ │ venue_name, venue_address │ │ city │ │ location (PostGIS POINT — lng, lat) │ └─────────────┘ │ date, time_start, time_end │ │ price, currency, external_url │ ┌─────────────┐ │ status (draft|active|cancelled|past) │ │ categories │────>│ views_count │ ├─────────────┤ └──────────────────────────────────────────┘ │ id (PK) │ │ name, slug │ Índices PostGIS: │ icon │ • GIST en location → O(log n) proximidad └─────────────┘ • B-tree en (status, date) → filtrado de feed

Consulta Clave: Eventos Cercanos

SELECT e.*, c.name AS category_name, ST_Distance(e.location, ST_MakePoint(lng, lat)::geography) AS distance_m FROM events e JOIN categories c ON c.id = e.category_id WHERE e.status = 'active' AND e.date >= CURRENT_DATE AND ST_DWithin(e.location, ST_MakePoint(lng, lat)::geography, 25000) ORDER BY distance_m ASC, e.date ASC LIMIT 20;

08 Progreso Actual y Roadmap

Next.js 15, Supabase, Tailwind CSS 4, pipeline de deploy en Vercel, TypeScript strict mode, path aliases, ESLint.

Email/contraseña + Google OAuth + modo invitado. Protección de rutas basada en proxy. Manejo de sesión vía cookies SSR de Supabase.

Renderizado en servidor con meta tags OG. Integración con mapa. Botón de compartir. Tracking de vistas.

Formulario de Subida de Flyer

Carga de imagen a Supabase Storage. Geocodificación de dirección. Selección de categoría. Selector de fecha/hora. Preview antes de publicar.

168 tests unitarios + 17 specs E2E. Vitest + Playwright configurados. Umbral de cobertura del 80% enforced.

Diseño dark minimalista siguiendo prototipo de Figma. Botones de Invitado, Iniciar Sesión y Google. En refinamiento.
